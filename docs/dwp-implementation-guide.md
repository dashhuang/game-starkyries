# "边玩边下"体验优化实践

> 基于《星河战姬 Starkyries》项目的实战经验。
>
> 本文不重复引擎 DWP 官方文档（`engine-docs/recipes/download-while-playing.md`）已涵盖的 API 用法和资源分类知识，而是聚焦于**官方行为之外、需要游戏侧主动补充的体验优化**。

---

## 为什么需要额外优化

引擎的 DWP 机制保证了"资源未到时不崩溃"——纹理显示占位色块、模型不渲染、音效静默、字体回退。但从玩家体验角度，这些**占位行为本身就是问题**：

| 资源类型 | 引擎默认的占位行为 | 玩家实际感受 |
|---------|-------------------|-------------|
| 纹理/图片 | 显示灰色方块，下载后跳变 | "这个游戏在卡" |
| 模型 | 不渲染，下载后突然出现 | "物体凭空冒出来了" |
| 音效 | 静默，错过播放时机 | 开枪没声音、击中没反馈 |
| 音乐 | 静默，下载后突然响起 | 进入场景几秒后音乐才冒出来 |
| 字体 | 回退字体，下载后跳变 | 文字突然变了样式 |
| 动画 | T-pose，下载后切换 | 角色先僵直再动起来 |

**核心观点：对于影响游戏体验的资源，不能依赖引擎的占位→热替换机制，必须确保在玩家用到之前就已经就绪。**

不同类型的资源，"用到之前"的含义不同，处理策略也不同：

| 资源特点 | 典型资源 | 策略 |
|---------|---------|------|
| **每帧渲染**，缺失可被看见 | UI 图标、立绘、背景图 | 骨架屏占位 + 门控 + 后台预下载 |
| **触发式**，缺失等于没有 | 音效、粒子特效 | 必须提前下载，无法补救 |
| **持续式**，延迟到达可感知 | BGM、环境音 | 场景切换前门控 |
| **结构性**，缺失导致画面异常 | 模型、动画、字体 | 必须提前下载，否则 T-pose/方块脸 |

下面按三层递进的思路，说明我们在项目中的具体做法。虽然以图片为主要案例（因为 UI 图标/立绘是本项目中数量最多的 DWP 资源），但**门控和后台预下载的机制对所有资源类型通用**。

## 一、逐帧安全加载 + 骨架屏占位

> 适用于 NanoVG 渲染的图片。其他资源类型（音效、模型等）不存在"骨架屏"概念，应直接走第二层门控。

### 问题

NanoVG 的 `nvgCreateImage()` 要求图片文件已在本地磁盘。在 DWP 下直接调用，要么拿到引擎占位纹理导致画面跳变，要么触发负缓存导致资源下载完也永远加载不出来。

### 做法

封装一个 `GetImage(nvg, path, cache, key)` 函数替代 `nvgCreateImage()`：

```
GetImage(path)
  ├─ 缓存命中？→ 直接返回句柄
  ├─ cache:Exists(path)？→ nvgCreateImage → 存缓存 → 返回句柄
  └─ 不在本地
      ├─ 缓存标记为 -2（下载中）
      ├─ cache:GetResourceAsync() 触发异步下载
      │   └─ 回调：清除缓存标记（不在回调里 nvgCreateImage，下帧重走 Exists 分支）
      └─ 返回 -2
```

调用方根据返回值决定渲染什么：

```lua
local img = ImageLoader.GetImage(nvg, path, imageCache, key)

if img > 0 then
    -- 正常渲染图片
elseif img == -2 then
    -- 下载中 → 显示骨架屏占位
    ImageLoader.RenderPlaceholder(nvg, x, y, w, h, animTime, cornerRadius)
end
```

### 骨架屏效果设计

所有"图片加载中"的区域使用**统一的占位效果**，避免各 UI 各写各的 fallback 导致视觉不一致。

效果：深色底板 + 脉冲呼吸（alpha 在 0.12~0.20 间缓慢波动）+ 从左到右的扫光。

设计原则：
- **不使用文字或加载圆弧**——骨架屏的意义是暗示"这里有内容"，而不是强调"正在加载"
- **颜色中性**（深灰蓝），适配各种 UI 风格
- **动画轻柔**（呼吸周期约 2.5 秒，扫光周期约 1.7 秒），不抢注意力
- **支持圆角**，与卡片、图标等各种形状匹配

### 一个关键设计决策：为什么回调里不直接 nvgCreateImage

异步下载回调的执行时机不在 NanoVG 的 BeginFrame/EndFrame 之间，直接操作 NanoVG 上下文有线程安全风险。所以回调里只**清除缓存标记**（设为 nil），让下一帧的渲染循环自然重走 `cache:Exists() → nvgCreateImage()` 的正常路径。代价是多等一帧，但安全可靠。

### 骨架屏是图片专属的兜底层

骨架屏只解决"图片还没到时不要显示灰块"这一个问题。但音效没到就是没声音，模型没到就是 T-pose——这些没法靠占位效果糊弄。所以对这些资源，必须在用到之前就保证就绪，也就是下面的门控和预下载。

## 二、界面/场景级加载门控

> **这是整个方案中最重要的一层**。对所有资源类型通用——图片、音效、音乐、模型、字体、动画都可以门控。

### 问题

有些界面打开时需要同时用到大量资源。如果全靠引擎的按需加载：
- 20+ 张武器图标逐个从灰块变成图片——满屏骨架屏逐个消失
- 战斗开始时 BGM 延迟 2 秒才响起——气氛全无
- 新关卡的敌人模型还在下载——玩家先挨了隐形敌人的打

### 做法

在进入界面/场景之前，先收集所有需要的资源路径，做一次**门控**检查：

```
进入界面/场景
  ├─ 收集所有需要的资源路径（图片、音效、模型、字体……）
  ├─ 逐个 cache:Exists() 检查
  ├─ 全部已缓存？→ 直接进入（零延迟快速路径）
  └─ 有未缓存的
      ├─ 弹出加载遮罩
      ├─ 逐个 GetResourceAsync 下载，每完成一个更新进度
      └─ 全部完成 → 关闭遮罩 → 进入
```

```lua
function WeaponSelectUI.Show(ship, callback)
    local paths = WeaponSelectUI.CollectImagePaths()
    ImageLoader.PreloadGate(paths, function()
        -- 走到这里时所有资源已就绪
        WeaponSelectUI.visible = true
    end, "正在加载武器数据...")
end
```

门控不限于图片——音效、模型、字体、BGM 都可以加入路径列表。`cache:Exists()` 对所有资源类型都有效。

### 哪些资源应该放进门控

判断原则：**如果这个资源缺失时玩家会明显感知到异常，就应该门控。**

| 资源 | 缺失时的表现 | 是否需要门控 |
|------|-------------|-------------|
| UI 界面的图标/立绘 | 灰块或骨架屏 | 是 |
| 战斗 BGM | 进入战斗后几秒无声 | 是 |
| 关键音效（开枪、爆炸） | 操作无反馈 | 是 |
| 敌人模型 | T-pose 或隐形 | 是 |
| 角色动画 | 先僵直再动 | 是 |
| 自定义字体 | 文字样式跳变 | 是 |
| 次要 UI 装饰图 | 细节缺失但不影响功能 | 可选（骨架屏兜底即可） |
| 环境音（鸟叫、风声） | 短暂缺失不明显 | 可选 |

### 加载遮罩的体验技巧

#### 技巧 1：最短显示时间

加载遮罩如果瞬间出现又消失（资源恰好刚下载完），用户会看到刺眼的一闪。**强制最短显示 0.3 秒**解决：

```lua
local minDisplayTime_ = 0.3

function LoadingOverlay._tryFinish()
    pendingFinish_ = true  -- 标记"想结束了"
    -- 实际结束在 Update 里判断 elapsed >= minDisplayTime
end

function LoadingOverlay.Update(dt)
    if not active_ then return end
    elapsed_ = elapsed_ + dt
    if pendingFinish_ and elapsed_ >= minDisplayTime_ then
        finishNow()
    end
end
```

0.3 秒是经验值——足够让用户意识到"加载了一下"，但不会觉得慢。

#### 技巧 2：手动模式精确进度

引擎提供 `ObserveDownloads()` 观察全局下载进度，但它的粒度是"所有正在下载的资源"，包括后台预加载的。我们需要的是"本界面这 20 个资源的进度"。

所以加载遮罩支持两种模式：

| 模式 | 进度来源 | 适用场景 |
|------|---------|---------|
| **自动模式** | `cache:ObserveDownloads()` | 进入关卡等大批量加载 |
| **手动模式** | 调用方每下完一个就 `SetProgress(n, total)` | PreloadGate 精确控制 |

PreloadGate 使用手动模式，可以精确显示"3/20"这样的进度，体验更好。

#### 技巧 3：回调叠加

如果遮罩已在显示中，又有新的 `Show()` 请求进来（例如用户快速切换界面），不重建遮罩，而是**叠加回调**：

```lua
if active_ then
    local prev = onReadyCallback_
    onReadyCallback_ = function()
        if prev then prev() end
        if onReady then onReady() end
    end
    return
end
```

#### 技巧 4：完成钩子

遮罩每次结束时调用所有注册的 `OnFinish` 钩子，用于通知其他系统（如后台预加载器恢复工作）。这是遮罩与后台预加载器之间的协作桥梁。

### 本项目使用门控的界面

| 界面 | 预加载的资源 | 状态文字 |
|------|------------|---------|
| 武器选择 | 所有武器图标 | "正在加载武器数据..." |
| 战舰选择 | 舰长立绘 | "正在加载战舰数据..." |
| 暂停菜单 | 武器 + 模块图标 | "正在加载暂停菜单..." |
| 图鉴 | 敌人 + 武器 + 模块图标 | "正在加载图鉴数据..." |
| 开箱 | 模块图标 | (默认文字) |
| GameOver | 舰长角色立绘 | "加载中..." |

### 门控 + 骨架屏 = 完整覆盖

门控保证 `Show()` 时刻核心资源就绪。但如果界面打开后又需要显示新资源（比如合成了新武器），骨架屏会自然兜底。两层配合才能覆盖所有情况。

对于不适合骨架屏的资源（音效、模型等），门控就是唯一的防线——所以门控的路径列表一定要**完整**，不要遗漏。

## 三、后台预下载调度

### 问题

即使有了门控，每次打开新界面仍可能触发加载遮罩——因为资源是首次请求时才开始下载。如果能利用空闲时间提前下载，用户打开界面时走的就是门控的"全部已缓存"快速路径，完全无感。

### 做法

在主菜单显示后启动后台预下载器，从游戏数据模块动态收集所有资源路径，按优先级排序后逐个串行下载。

```
Start()
  ├─ 从配置模块收集所有资源路径（图片、音效、模型……）
  ├─ 按优先级排序 + 全局去重
  ├─ CanResolve() 过滤（跳过构建引用中不存在的路径）
  └─ DownloadNext() 串行下载
      ├─ cache:Exists？→ 跳过
      ├─ LoadingOverlay.IsActive()？→ 暂停
      └─ cache:GetResourceAsync() → 完成后继续下一个
```

### 优先级设计

优先级的核心原则：**按玩家流程中可能最先遇到的顺序排列**。

以本项目为例：

| 优先级 | 资源类型 | 理由 |
|--------|---------|------|
| 1 | 开场教程对话图片 | 新玩家最先看到 |
| 2 | 舰长立绘 | 选船界面必需 |
| 3 | 武器图标 | 选武器、商店、暂停菜单多处复用 |
| 4 | 模块图标 | 商店、暂停菜单、开箱多处复用 |
| 5 | 首次战败对话图片 | 新玩家较早触发 |
| 6 | 默认舰长战败图 | 初始舰长的 GameOver 场景 |
| 7 | 其余所有对话图片 | 低优先级，空闲时下载 |
| 末 | 非默认舰长战败图 | 解锁新舰长后才需要 |

设计优先级时的思考方式：
- **从新玩家视角出发**，他最先进入什么界面、最先触发什么事件？
- **高复用资源优先**：一个资源在多个界面用到，优先级应该更高
- **解锁后才需要的资源排最后**：非默认角色、高级关卡的资源

### 与前台加载的协作

后台下载必须给前台让路，否则会抢占带宽导致前台加载变慢。

机制：
1. 每次取下一个资源前，检查 `LoadingOverlay.IsActive()`
2. 如果前台在加载 → 暂停，设 `paused = true`
3. 注册 `LoadingOverlay.OnFinish()` 钩子
4. 前台加载结束 → 钩子触发 `Resume()` → 继续后台下载

```
前台 PreloadGate 触发
  → LoadingOverlay 显示
  → BackgroundPreloader 检测到 IsActive()
  → 自动暂停

前台加载完成
  → LoadingOverlay 结束
  → 触发 OnFinish 钩子
  → BackgroundPreloader.Resume()
  → 继续后台下载
```

### 为什么用串行而非并行

串行下载（一次只下一个）的好处：
- **可预测**：明确知道当前在下第几个
- **可暂停**：前台需要让路时立即生效，不用等一批并发请求完成
- **低开销**：不会因为并发请求过多影响游戏帧率
- **够快**：带宽瓶颈在网络，串行和并行的总时间差别不大

### 调试手段

后台预下载默认静默运行，不输出任何信息、不显示任何面板。通过测试菜单的"预加载调试"开关可打开：

- **控制台输出**：每个资源的下载/跳过/失败日志
- **屏幕面板**：右下角实时显示进度、当前文件、统计数据

```lua
BackgroundPreloader.SetVerbose(true)   -- 开启
BackgroundPreloader.SetVerbose(false)  -- 关闭
```

设计为开关而非默认打印，是因为正式环境下不能干扰玩家，但开发调试时需要确认：
- 优先级顺序是否合理（最先下的是否确实是玩家最先用到的）
- 是否有资源被遗漏（打开界面仍然触发了加载遮罩）
- 前台/后台的协作是否正常（暂停和恢复的时机是否正确）

## 新项目复用建议

### 按项目规模选择方案

| 规模 | 建议 |
|------|------|
| 图片少、无复杂 UI | 只做第一层（骨架屏兜底），或直接门控 |
| 有多个 UI 界面 | 第一层 + 第二层（门控 + 加载遮罩） |
| 资源非常多、首屏体验要求高 | 三层全上，重点设计优先级 |

### 适配新界面/场景的标准步骤

```
1. 列出该界面/场景用到的所有媒体资源（图片、音效、模型、字体……）
2. 判断哪些缺失时玩家会明显感知 → 这些是门控必须覆盖的
3. 在 Show() / Enter() 入口加 PreloadGate，传入路径列表
4. 如果是 NanoVG 图片 → 渲染时用 GetImage + RenderPlaceholder 兜底
5. 把高频使用的路径加入后台预下载器的优先级队列
```

## 踩过的坑

### NanoVG 图片句柄泄漏

`nvgCreateImage()` 每次调用都分配 GPU 显存。如果在每帧渲染中反复调用而不缓存，会导致显存泄漏。必须把句柄存到缓存表里复用。

### 异步回调中不要操作 NanoVG

下载完成的回调不在渲染帧内，直接调 `nvgCreateImage()` 有线程安全风险。正确做法是回调里只清标记，让渲染循环下一帧处理。

### 加载遮罩闪烁

遮罩瞬间出现又消失比不显示更刺眼。强制最短 0.3 秒显示。

### 后台下载抢带宽

后台预加载器不检查前台状态就一路下载，会导致前台加载遮罩显示时间变长。必须有暂停/恢复机制。

### nvgCreateFont 不要每帧调用

`nvgCreateFont()` 也会分配资源，必须只在初始化时调用一次。容易在 DWP 适配过程中误加到每帧逻辑里。

### 门控路径列表不完整

音效、模型等资源忘记加入门控列表，导致进入战斗时没声音、敌人模型延迟出现。门控的路径列表一定要完整覆盖所有关键资源，不能只关注图片。
